# Flat Hunter
> **⚠️ Внимание:** Активная разработка, исходный код и CI/CD пайплайны этого проекта находятся на **GitLab**: [https://gitlab.com/sh1n1ta1d3su/flat_hunt/].
> 
> **Почему GitLab?** Я намеренно усложнил этот пет-проект, чтобы отработать навыки DevOps.Здесь реализована микросервисная архитектура (Celery, Redis, PostgreSQL), код автоматически проверяется линтерами, проходит Unit-тесты и собирается в готовые Docker-образы через GitLab CI/CD (Docker-in-Docker). Данный репозиторий на GitHub служит зеркалом и портфолио.

Автоматизированная микросервисная система непрерывного мониторинга сайтов аренда недвижимости
Проект парсит объявления (в данный момент реализована поддержка Яндекс.Недвижимости), сохраняет новые квартиры в базу данных во избежание дубликатов и мгновенно отправляет уведомления пользователю в Telegram.

## 🛠 Стек
* **Язык:** Python 3.11
* **Парсинг:** `requests`, `beautifulsoup4`
* **Telegram Bot:** `aiogram` (Long-polling для команд + сессии для воркера)
* **База данных:** PostgreSQL, `psycopg2-binary`
* **Очереди задач:** Celery, Redis
* **DevOps & CI/CD:** Docker, Docker Compose, GitLab CI/CD, GitLab Container Registry, Docker-in-Docker (dind)
* **Тестирование и Качество кода:** `pytest`, `flake8`

## 🏗 Эволюция архитектуры проекта
Flat Hunter v1.0 (Монолит)
Изначально проект запускался в двух контейнерах ('bot' и 'db') и работал как монолит. Парсер функционировал в бесконечном цикле 'while True' с 'time.sleep()'.
Проблема: Блокирующий цикл мешал масштабированию системы, добавлению новых сайтов-доноров и оптимальному использованию ресурсов.

Flat Hunter v2.0 (Текущая версия)
Был произведен полный рефакторинг с переходом на асинхронные очереди задач. Теперь система масштабируема, отказоустойчива и разделена на изолированные роли в рамках 'docker-compose'.

---

## ⚙️ Роли микросервисов (Docker Compose)
Система разделена на 5 независимых контейнера:

1. **База данных ('db' / PostgreSQL):** Хранит историю обработанных объявлений по уникальным ссылкам.
2. **Брокер сообщений ('redis'):** Хранит очередь задач на парсинг.
3. **Планировщик ('celery_beat'):** Менеджер, который по расписанию генерирует задачу «спарсить сайт» и кладет её в Redis.
4. **Рабочий узел ('celery_worker'):** Непрерывно слушает очередь Redis. Как только появляется задача, он скачивает HTML, парсит объявления, проверяет их по БД и дергает логику отправки Telegram-уведомлений.
5. **Интерактивный интерфейс (`bot`):** Асинхронный демон, который слушает команды пользователя в Telegram и позволяет управлять логикой парсера без перезапуска контейнеров.

## 🎮 Управление ботом

Система спроектирована под индивидуальное использование и реагирует только на команды владельца (проверка по Telegram User ID):

- `/set <ссылка>` — Перезаписать поисковый фильтр. Старая ссылка удаляется из базы, и система мгновенно переключается на мониторинг новых критериев (например, изменился бюджет или район).
- `/current` — Показать активную ссылку, по которой в данный момент работает парсер.
- `/start` — Приветственное сообщение и краткая справка.

## Быстрый старт
1. Клонирование репозитория
```bash
git clone https://github.com/ТВОЙ_НИК/flat_hunter.git
cd flat_hunter
```

---

2. Настройка окружения 
Создайте файл '.env' в корне проекта и заполните его по примеру:
```
# Токен вашего Telegram-бота (получить у @BotFather)
BOT_TOKEN=your_telegram_bot_token
# Ваш Telegram ID (кому бот будет присылать уведомления)
TELEGRAM_USER_ID=987654321
    
# Настройки Базы Данных
DB_HOST=db
POSTGRES_USER=hunter
POSTGRES_PASSWORD=hunter_password
POSTGRES_DB=flathunter

# Настройки Celery / Redis
REDIS_URL=redis://redis:6379/0
```

---

3. Запуск системы
```bash
docker compose up -d --build
```

---

4. Мониторинг и логи

Чтобы проверить, как работает система и убедиться, что парсер получает задачи:

- Логи рабочего воркера (парсинг): 'docker compose logs -f celery_worker'

- Логи планировщика: 'docker compose logs -f celery_beat'

- Логи базы данных: 'docker compose logs -f db'
